# Security & Privacy Governance（Token、日誌、資料最小化）

> 本文件為 MVP 上線前的安全與隱私基線，適用於 iOS App、後端 API 與 GitHub 自動化流程。

## 1. Token 安全策略（加密、到期刷新、撤銷）

### 1.1 儲存原則
- Access Token、Refresh Token、API Key 一律存放在受管 Secrets（如 GitHub Secrets、部署平台 Secrets、行動端安全儲存）。
- 禁止將 Token 寫入：
  - Git tracked 檔案（程式碼、文件、測試樣本）
  - Issue / PR 內容
  - 應用程式與 CI 日誌

### 1.2 加密原則
- Token 在儲存時必須使用平台提供的加密機制（at-rest encryption）。
- 需要落地保存的憑證資料，必須使用加密儲存，不得明文保存。

### 1.3 到期刷新與失效管理
- Access Token 採短時效策略（依 OAuth Provider 規範）。
- 到期前或收到 401/invalid_token 時觸發 refresh flow。
- Refresh Token 若支援 rotation，應於刷新成功後立即替換舊值。
- 帳號登出、權限撤銷、疑似外洩時，必須清除本地 token 並觸發重新授權。

## 2. 日誌去識別化（De-identification）

### 2.1 最小記錄原則
- 日誌僅記錄除錯與稽核必要資訊（事件時間、模組、錯誤代碼、追蹤 ID）。
- 郵件內文、完整信箱地址、Token、Authorization Header、Cookie 不得寫入日誌。

### 2.2 遮罩與匿名化規則
- Email：保留網域或部分前綴（例如 `u***@example.com`）。
- Token / 金鑰：一律以 `[REDACTED]` 取代；若需追蹤關聯，改用不可逆雜湊值。
- 使用者識別值：優先使用不可逆雜湊或系統內部追蹤 ID。

### 2.3 稽核規則
- PR 合併前需確認未新增任何敏感樣本資料。
- 發版前需抽查應用與 CI 日誌，確認無敏感欄位外露。
- 程式化去識別化請使用 `src/log-redaction.js` 的 `sanitizeLogValue`，避免直接輸出原始 payload。

## 3. 第三方服務資料外送告知與同意

### 3.1 告知要求
- 對外傳輸資料前（例如翻譯 API），需明確告知：
  - 傳輸目的（如翻譯英文郵件）
  - 傳輸資料範圍（如郵件主旨/內文片段）
  - 接收方（第三方服務名稱）

### 3.2 同意要求
- 需有明確同意動作（按鈕觸發或勾選同意），不得預設自動外送。
- 使用者可取消或停止該功能，且不影響核心收信能力。

### 3.3 文件化紀錄
- 於需求文件或 Issue 中記錄每一項第三方資料流：
  - 資料欄位
  - 使用目的
  - 保存期限
  - 使用者同意方式

| 資料流 | 資料欄位 | 使用目的 | 保存期限 | 同意方式 |
| --- | --- | --- | --- | --- |
| 翻譯服務（LibreTranslate） | 郵件內文 `body`（使用者按鈕觸發後） | 英文翻譯為繁中 | 不在本地額外持久化翻譯請求 payload；僅保留當前顯示狀態 | 使用者每次手動點擊「翻譯」按鈕即視為當次明確同意（傳輸前不改寫原文，僅在 log 層以 `sanitizeLogValue` 去識別化） |
